# S01: QR identity and local fridge entry

**Goal:** Establish the first real product loop by giving fridges/freezers stable identities, printable QR codes, and a local web entry flow that opens the correct storage context.
**Demo:** You can create a printable QR code for a fridge/freezer, scan it on the home network, and land in that storage context in the local web app.

## Must-Haves
- A local web app entrypoint exists and can run in development.
- The system can create fridge/freezer identity records with stable IDs.
- Each storage context can render a printable QR code generated by the app itself.
- Scanning or opening the QR target URL resolves the correct fridge/freezer context.
- The storage-context page shows enough information to prove the right fridge/freezer was loaded.
- Basic failure states exist for missing or invalid QR/context IDs.

## Observability / Diagnostics

**Runtime signals:**
- The Next.js dev server logs all route hits to stdout; check terminal output to confirm routes are resolved.
- The fridge-context page (`/fridges/[fridgeId]`) must render the fridge name and storage type so any agent can visually confirm the right record loaded.
- API routes under `/api/fridges` return structured JSON with an `error` field on failure; log these in the browser network panel.
- SQLite database file lives at `data/fridges.db`; inspect with `sqlite3 data/fridges.db .dump` to verify records.
- QR generation errors surface via `console.error` in the QR component and appear in browser DevTools console.

**Failure visibility:**
- Invalid or missing `fridgeId` renders a `not-found` page with a clear message — no silent blank screens.
- Database write failures throw and are surfaced in the terminal as uncaught Next.js server errors.
- QR code generation falls back to a text URL display when the canvas/SVG render fails.

**Redaction:** No PII beyond fridge/freezer names (user-supplied) is stored; names are not redacted in dev logs.

**Inspection commands:**
```bash
# Confirm dev server is healthy
curl -s http://localhost:3000 | grep -c "theFridge"

# Check a storage-context route
curl -s http://localhost:3000/fridges/<id> | grep -c "storage"

# Inspect the database
sqlite3 data/fridges.db ".tables"
```

## Verification

```bash
# 1. Dev server boots without errors
npm run dev > /tmp/devserver.log 2>&1 &
sleep 5 && curl -s -o /dev/null -w "%{http_code}" http://localhost:3000

# 2. Home page returns 200 with app title
curl -s http://localhost:3000 | grep -c "theFridge"

# 3. Dynamic fridge route returns 200 (with a real id) or handled 404
curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/fridges/does-not-exist

# 4. TypeScript compiles without errors
npx tsc --noEmit

# 5. Database tables exist after first boot (diagnostic failure path check)
sqlite3 data/fridges.db ".tables" 2>/dev/null | grep -c "fridge" || echo "db-not-created-yet"

# 6. Invalid ID renders failure state (not a blank page or uncaught exception)
curl -s http://localhost:3000/fridges/does-not-exist | grep -c "not found\|STORAGE NOT FOUND"

# 7. Valid fridge ID renders identity card with fridge name
FRIDGE_ID=$(sqlite3 data/fridges.db "SELECT id FROM fridges LIMIT 1;" 2>/dev/null) && \
  [ -n "$FRIDGE_ID" ] && curl -s "http://localhost:3000/fridges/$FRIDGE_ID" | grep -c "storage context\|STORAGE CONTEXT" || echo "no-fridge-records-yet"
```
