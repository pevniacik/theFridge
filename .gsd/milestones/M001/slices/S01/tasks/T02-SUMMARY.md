# T02: Implement fridge/freezer identity records and QR generation

**Wired the full identity loop: SQLite persistence, nanoid IDs, Server Action create flow, REST API, and server-side SVG QR code generation — creating a fridge redirects directly to its context page with a printable QR code already rendered.**

## What Happened

Built the data layer from scratch: `lib/db/client.ts` provides a singleton `better-sqlite3` connection that creates `data/fridges.db` and applies the schema (`fridges` table) on first access. `lib/fridges/store.ts` implements `createFridge`, `getFridgeById`, and `listFridges` — all synchronous, matching the better-sqlite3 API.

QR generation lives in `lib/qr/generate.ts`: builds the fridge URL from a caller-supplied base URL (derived from request headers so it works on LAN IPs), calls `QRCode.toString(..., { type: 'svg' })`, and returns either `{ svg, url }` or `{ error }` — never throws, so the UI always has a fallback path.

The create flow uses a Server Action (`app/fridges/new/actions.ts`) with the `(prevState, formData)` signature required by `useActionState`. On success it calls `redirect()` which throws internally and routes the user to `/fridges/<id>`. On validation failure it returns `{ error: "..." }` which the client form displays inline.

`app/api/fridges/route.ts` provides GET (list) and POST (create) as a REST surface — useful for programmatic access and curl-based inspection.

The fridge context page (`app/fridges/[fridgeId]/page.tsx`) was updated to call `getFridgeById` from the store. Valid IDs get the full identity card + QR code panel + inventory placeholder. Invalid IDs get a clear "STORAGE NOT FOUND" UI with actionable links — no silent blank screen.

One TypeScript issue was caught and fixed before the dev server started: the `useActionState` overload requires the action to accept `(prevState: S, formData: FormData)`, not just `(formData: FormData)`.

## Diagnostics

- **DB health:** `sqlite3 data/fridges.db ".tables"` → should print `fridges`; `sqlite3 data/fridges.db "SELECT * FROM fridges;"` → lists all identity records
- **API inspection:** `curl http://localhost:3000/api/fridges` → JSON list; error responses always have `{ error: "..." }` field at 4xx/5xx
- **QR failures:** appear as `[qr] Failed to generate QR code for <id>: <message>` in the Next.js server terminal; UI falls back to a text URL link
- **DB write failures:** throw from `createFridge` / `store.ts`, surface in Next.js server log as uncaught errors with full stack
- **Server log signal:** `GET /fridges/<id> 200 in Xms` confirms identity resolution; `GET /fridges/<unknown-id> 200 in Xms` with "not found" content is normal (not a 404)

## Deviations

- **useActionState signature fix** — the plan described the Server Action as `(formData: FormData) => ...` but `useActionState` requires `(prevState: S, formData: FormData) => ...`. Updated action signature and exported `CreateFridgeState` type; this is a React 19 / Next.js 15 server actions contract, not a plan mistake.
- **Not-found rendered inline** — used a conditional render inside the context page rather than Next.js `notFound()`. This gives a richer "STORAGE NOT FOUND" card with actionable links rather than the generic 404 page.

## Known Issues

- The `data/fridges.db` is not gitignored (intentionally — it seeds real data locally). Add to `.gitignore` if this ever becomes a shared repo with a proper migration strategy.
