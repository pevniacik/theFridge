# T02: Wire Serwist service worker, offline page, and tsconfig update

**Wired Serwist into the Next.js build pipeline: `next.config.ts` wrapped with `withSerwistInit`, `app/sw.ts` service worker created, `/~offline` fallback page added, and `public/sw.js` (42KB) generated by production build.**

## What Happened

Implemented all four file changes required to integrate Serwist into the build pipeline:

1. **`next.config.ts`** — Wrapped the existing `output: "standalone"` config with `withSerwistInit({ swSrc: "app/sw.ts", swDest: "public/sw.js", disable: process.env.NODE_ENV === "development", additionalPrecacheEntries: [{ url: "/~offline", revision }] })`. Revision is computed via `git rev-parse HEAD` with `crypto.randomUUID()` as fallback.

2. **`app/sw.ts`** — Created Serwist service worker with triple-slash `/// <reference no-default-lib="true"/>`, `/// <reference lib="esnext" />`, `/// <reference lib="webworker" />`. Uses `defaultCache` from `@serwist/next/worker`, declares `self.__SW_MANIFEST`, instantiates `Serwist` with `precacheEntries`, `skipWaiting: true`, `clientsClaim: true`, `navigationPreload: true`, and offline fallback for document requests routing to `/~offline`. Calls `serwist.addEventListeners()`.

3. **`app/~offline/page.tsx`** — Static React component with centered layout using CSS variables (`var(--color-bg)`, `var(--color-text)`, `var(--color-text-muted)`), a 🧊 icon, and descriptive message matching app style.

4. **`tsconfig.json` + `tsconfig.worker.json`** — **Key deviation from plan:** The plan said to add `"webworker"` to the main tsconfig's `lib` array, but this breaks all DOM component types (EventTarget, Navigator, Window, etc.). Instead, `app/sw.ts` is added to the `exclude` array in `tsconfig.json`, and a new `tsconfig.worker.json` is created that extends the main config but overrides `lib: ["esnext", "webworker"]` and only includes `app/sw.ts`.

- **Main types:** `npm run type-check` — verifies DOM components still compile after SW exclusion
- **Build failure surface:** Serwist emits `[serwist]`-prefixed errors; TypeScript errors from SW bleeding into main compilation manifest as `EventTarget & HTMLElement` losing DOM properties

## Deviations

**tsconfig approach changed:** Plan said add `"webworker"` to `lib` in `tsconfig.json`. Adding it globally removes DOM-only types and breaks all component event handlers (`e.target.files`, `e.target.value`, `navigator.clipboard`, etc.). Instead: excluded `app/sw.ts` from the main tsconfig and created `tsconfig.worker.json` for isolated worker type-checking. This is the standard Next.js + Serwist pattern — the triple-slash directives in `app/sw.ts` are sufficient for Webpack to type it correctly during the build; the separate tsconfig is for editor/CI type-checking only.
